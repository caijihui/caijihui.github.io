大报文接口导致fullGC

发表于 2024-09-15

频繁gc排查

监控表象:
近三小时调用4000次,耗时 > 1s 80次 ,耗时 >3s 27次
服务频繁重启,且qps 并不高.

具体分析

现象1: http 调用耗时高，且超时

  • 原因
    • 大报文导致 传输出现超时
  • 解决方案
    • 配置http超时时间
    • httpclient3 连接池
    • 大报文处理
      • 禁止大报文数据的拷贝
      • 禁止大报文数据 log打印
      • 拆分接口

现象2: ClientAbortException

异常通常发生在一个客户端关闭了与服务器的连接，而服务器仍然尝试向这个客户端写数据时。

  • 原因
    • 由于超时或者报文太大，长时间获取不到返回，客户端断开了链接。
  • 解决方案
    • 报文设置是否合理
    • 配置超时时间

现象3: 设置缓存耗时较长，1.16s

bigkey 设置缓慢

  • 原因

    • key 对应的 value 比较大，俗称bigkey。

  • 解决方案:

    • 避免bigkey
    • 拆分成小key
    • 异步删除
    • lru 淘汰策略

bigkey 的主要影响

  • 网络阻塞，获取 bigkey 时，传输的数据量比较大，会增加带宽的压力。
  • 超时阻塞; 因为 bigkey 占用的空间比较大，所以操作起来效率会比较低，导致出现阻塞的可能性增加。
  • 导致内存空间不平衡: 一个bigkey 存储数据量比较大，同一个key 在同一个节点或服务器中存储会造成一定影响。

服务器容量预估和QPS计算

发表于 2024-03-15 更新于 2024-09-15

jvm堆内存设置

堆内存: 默认物理内存为1/4
Xms512m -Xmx1024m
最大常规做法: Xms=Xmx=物理内存的1/2 Xmn=1/2Xmx

通过jvm内存 预估服务的QPS

1核8G
单个请求15kb，每分钟就是 15*60=900kb约 0.9M 约1M

1/4 内存设置堆内存为2G:
2048/3=682.667M 新生代内存新生代 ，老年代 1: 2
682.667/0.9=758.519 qps

700个请求: 700*1M = 700M

1G机器可支持大约为700 qps

峰值QPS：

百分之80的流量，发生在20%的时间内。

峰值QPS:单日总pv* 80%/20% 243600

实际业务计算QPS

QPS计算：
一分钟，15542次 ，即 260 /s
TPS 100/s，一般qps 是tps 的3倍。

预估CPU = QPS / 100
预估内存 = QPS /1002
预估磁盘 = QPS / 1002
预估网络 = QPS /1002
预估操作系统 = QPS /1002
预估数据库 = QPS / 100 *2

以QPS 300 为例测算

预估cpu 300/100= 向上为4核
预估内存 300/100*2=6G ，此刻jvm 内存可设置为3G，当然要想支持更大的QPS，也可以通过加机器，加pod的方式扩展。
一台qps300，两台就是600l。
由于每个服务的业务复杂度不尽相同，这里只能预估一个大概的配置。
